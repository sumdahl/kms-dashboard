# Fix Compilation Errors in KMS Dashboard

## Problem Analysis
The project has 82 compilation errors related to:
1. Template parsing errors in Askama templates (using Rust `if let` syntax instead of Jinja2-style conditionals)
2. Missing `IntoResponse` implementations for template structs
3. Template syntax issues causing parser failures

## Root Causes
- Templates use `{% if let Some(msg) = variable %}` which may not be properly supported in Askama 0.12
- Template structs call `.into_response()` directly instead of using the provided `render()` function
- Earlier syntax errors may cause cascading parser failures (like `elif` not being recognized replace with else if)

## Solution Plan

### Phase 1: Fix Template Syntax
Update all templates to use correct Askama syntax for Option handling:
- Replace `{% if let Some(msg) = variable %}{{ fb::alert(message=msg, ...) }}{% endif %}`
  with `{% if variable %}{{ fb::alert(message=variable, ...) }}{% endif %}`
- Askama automatically handles Option<T> by rendering the inner value when Some, or nothing when None
- This avoids potential issues with `if let` destructuring in template scope

### Phase 2: Fix Handler Code
Replace direct `.into_response()` calls on templates with `render(template)` calls:
- Update all handlers to use `crate::render::render(template)` instead of `template.into_response()`
- Remove unused imports and add necessary imports

### Phase 3: Verify and Test
- Run `cargo build` to ensure all errors are resolved
- Run `cargo check` for additional validation
- Test that templates render correctly

## Implementation Steps

1. **Update template syntax in all .html files**
   - Find all instances of `{% if let Some(msg) = variable %}...{{ fb::alert(message=msg, ...) }}...{% endif %}`
   - Replace with `{% if variable %}...{{ fb::alert(message=variable, ...) }}...{% endif %}`
   - This leverages Askama's built-in Option handling where Option fields render their inner value when Some

2. **Update Rust handler code**
   - Replace `template.into_response()` with `render(template)`
   - Add `use crate::render::render;` where needed
   - Remove direct `IntoResponse` calls on templates

3. **Clean up imports**
   - Remove unused imports flagged by warnings
   - Ensure all necessary imports are present
